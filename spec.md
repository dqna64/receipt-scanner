# The Project: A personal receipt scanning web app

This project is a receipt scanning app for personal use.

# Project ground rules

We will design it to suit my own requirements as the (currently only) user, while building it as a genuine multi-tenant SaaS: every signup is a first-class, equal peer — no privileged "owner" account, no first-user special-casing. All per-user data (receipts, items, categories, tags, payment methods, recurring defs, sessions) is scoped by user_id with fully isolated access and no cross-account interactions. "Scalability best practices without the cost of scaling" refers to INFRASTRUCTURE (one box, no horizontal scaling, no Redis/queue) — NOT to the tenancy model, which is real multi-user from day one. Being the only user today is incidental; nothing in the design assumes it.

# Features & User Stories (M1)

Currently one user ("I"), but every story is written for a multi-tenant SaaS — any signup gets the same isolated experience. Stories marked (open) depend on decisions still being discussed.

## Auth & accounts
- Anyone can register an account (email + password), self-service, the way any SaaS works — all signups are equal peers, no first-user privilege, each gets its own fully isolated tenant. Open registration is guarded by ordinary SaaS abuse controls, NOT a bootstrap token: rate limiting (decided) + email verification (decided as REQUIRED before public open signup — see Auth in Architecture; gates spam accounts and, critically, LLM-cost abuse since each account can spend scan money). Registration seeds that new user's own default categories + "cash" payment method (per-user, at signup — see Auth).
- I can log in and stay logged in on my phone and laptop (session cookie, ~30-day sliding expiry) and log out. Self-service password reset (decided, needs email — same infra as verification).
- Login endpoint is rate-limited; the registration endpoint is rate-limited too (open signup on a public host — throttles automated account/cost abuse).

## Capture
- I can upload one or more receipt photos from my phone camera or a file picker. Client re-encodes/downscales before upload (handles HEIC, cuts vision-token cost).
- Each image immediately creates a receipt row (scan_state=uploaded) — nothing is ever lost in a waiting state. I see it appear in history with a live status trace: uploaded → scanning → pending review (or scan failed). (That user-facing trace is the scan_state lane plus the derived needs_review state; see Invariants.)
- On scan failure I can retry the scan or fall back to entering the fields manually against the stored image.
- I can record a cash/no-receipt transaction via a manual entry form (merchant, date, total, currency, payment method defaulting to cash, category, tax-deductible, note, optional items). Manual entries are born with reviewed_at set (proposed_at NULL) so they count immediately — I typed them, no review needed.
- Quick-add (required by the <15s friction ceiling): the manual form defaults everything defaultable (date=today, currency=AUD, payment=cash) so a coffee is amount + merchant + category and done; recently-used merchants autocomplete.
- I can record a refund/return: scan the return docket (they're receipts) or manual entry; kind=refund (inflow), optional link to the original receipt.
- I can record a reimbursement I receive (PayID/Beem etc.): a manual entry, kind=reimbursement (inflow). These usually arrive with no purchase context — I annotate which expense they offset (same category as the original so category charts net correctly; optional link).
- I can record an ATM withdrawal as a transfer — visible for cash reconciliation, never counted as spend (the individual cash purchases are the spend).
- I can record INCOME (decided in M1): scan a paycheck/payslip (kind-aware scan classifies it as income and extracts employer + pay date + NET amount — see Scan pipeline), or enter income manually (salary, casual/gig pay, bank interest, distributions/dividends, a friend gifting me money). Income is kind=income (inflow), carries an income category (salary/interest/dividends/other-income — the category picker filters to income categories when kind=income), does NOT count as spend, and feeds income stats + the credit side of reconciliation. Manual income is born reviewed (counts immediately); a scanned paycheck goes through review like any scan. Recurring income (fortnightly salary, monthly interest) is supported via a recurring definition with kind=income (see Recurring).

## Transaction kinds & stats rules (decided)
- receipts.kind: purchase (default) | refund | reimbursement | income | transfer. `income` is now IN M1 (decided 2026-07-18 — salary/paychecks, casual income, bank interest, distributions/dividends; scanned from payslips or manually entered). The enum stays extensible.
- Magnitude + direction money model (decided 2026-07-18 — replaces the old signed-amount convention): `total_cents` (and item `amount_cents`) are stored as a NON-NEGATIVE MAGNITUDE (≥ 0), never signed. Physical cash direction is carried EXPLICITLY in a `direction` field [inflow | outflow] (= bank credit vs debit), NOT as the sign of the amount. Stats apply per-kind signs in the query/view, not in storage. Rationale: income made "negative = money in" unintuitive, and transfers can go EITHER direction (ATM withdrawal = outflow, share-sale proceeds = inflow) so direction can't be derived from kind alone. `direction` is the reconciliation key (which bank-statement side a row lands on); `kind` is the semantic role. For the fixed kinds a CHECK enforces the canonical direction (purchase → outflow; refund/reimbursement/income → inflow); transfer sets direction freely. ("transfer" is a KIND, not a direction value — a transfer still has an inflow/outflow direction; keeping direction a clean two-value field avoids a redundant tri-state.)
- Kind → stats role:
    - purchase: outflow, +spend.
    - refund: inflow, −spend (nets against spend, same category — e.g. return docket, cashback/reward credit categorized "rewards").
    - reimbursement: inflow, −spend (nets against the specific expense it offsets; annotate which — same category as the original; optional link). A Beem/PayID repayment for something you fronted is a reimbursement.
    - income: inflow, does NOT touch spend — it's a separate axis, summed into INCOME stats, and feeds the CREDIT side of reconciliation. (A friend GIFTING you money with no expense to offset is income, not reimbursement.)
    - transfer: own-money movement, excluded from BOTH spend and income. Covers ATM withdrawals, credit-card bill payments (the card's purchases were already counted), moves to savings/investments (incl. after-tax personal super contributions — see super boundary), loan/BNPL principal repayments (the interest/fee portion is a real purchase, category "fees"), and the RETURN direction too (selling shares, withdrawing from savings = inflow transfer).
- Spend stats = Σ(magnitude with per-kind sign) over kinds {purchase (+), refund (−), reimbursement (−)} ONLY — NOT "≠ transfer" (that would now wrongly include income) — further excluding purchases paid with stored-value methods. Income stats = Σ(income magnitude). Transfers in neither. (The per-kind sign lives in the stats VIEW, single-sourced — see schema.)
- Expenses incurred for others still count in full as my expense; incoming reimbursements offset them as inflow reimbursement entries.
- Refundable deposits/bonds (rental bond, hire deposits): purchase (category "bonds/deposits") + linked refund when returned. If the bond is docked, the books are already correct.
- Stored-value convention: gift card/Opal/prepaid TOP-UPS count as the expense at top-up time (kind=purchase, categorized e.g. "transport", "gifts"). Purchases PAID WITH a stored-value method are recorded (receipt, items, history, dedupe) but excluded from spend stats — the money was already counted. Requires payment methods to carry a stored_value flag.
- Cash is the opposite: ATM withdrawal (kind=transfer, outflow) excluded; each cash purchase counts at spend time.
- The stored-value convention is chosen PER payment method via the stored_value flag: flag an Opal/gift card (spend books at top-up), leave a travel money card (Wise) unflagged and record its loads as transfers to keep per-purchase categories abroad.
- Super boundary (decided 2026-07-18): record ONLY what touches my bank account. Net pay = income (inflow); after-tax personal super contributions = transfer (outflow to a retirement asset). Employer SG and salary-sacrifice super never hit my bank (employer → fund, or deducted pre-tax from gross) and are OUT OF SCOPE — this app reconciles against my bank account, so gross-salary → PAYG-tax → super → net decomposition is payroll accounting, not spend/income tracking. Paycheck scanning therefore records the NET pay as the income amount (the payslip's gross/tax/super lines may be captured as informational best-effort items but never feed stats or reconciliation).
- Investments record the CASH MOVEMENT only, not the asset (decided 2026-07-18): buying shares/ETFs = transfer (outflow); selling = transfer (inflow); brokerage fee = a separate purchase, category "fees". Units held, cost basis, and market valuation are NOT represented — portfolio/holdings tracking is a different domain and explicitly out of scope. Dividends/distributions received in cash = income; a DRP (reinvested distribution) = income (inflow) immediately followed by a transfer (outflow) if you want both legs, or just skip the wash if you only track cash.
- Bank interest: interest the bank PAYS me on savings = income (inflow, income category "interest"); interest CHARGED to me (loan/credit-card) = purchase, category "fees" (outflow). Direction disambiguates the two.
- Small conventions: pre-auth holds ($1 petrol, hotel holds) are not transactions — reconcile settled lines only; cashback/reward credits = refund (inflow) with category "rewards" (or ignore); bank/account fees, fines, ATO payments = ordinary manual purchases; zero-total receipts (warranty swaps) are allowed.

## Review
- I have a "pending review" queue (oldest first, with a count badge) of everything machine-generated: scanned receipts and auto-created recurring charges.
- The review screen shows the receipt image alongside the extracted fields; I can correct merchant, date, total, currency, payment method, category, tax-deductible, note, and line items, then mark reviewed.
- Line items are extracted and auto-categorized by the scanner but OPTIONAL to review (decided): marking reviewed only requires confirming receipt-level fields — a 40-item grocery shop is not a chore. Item-level data is best-effort: as good as the auto-labels plus whatever I choose to correct.
- If line items don't sum to the total, I get a warning, not a block (discounts/rounding are real).
- Categorization (decided): one user-editable category taxonomy (seeded with sensible defaults), used at both receipt level ("groceries") and item level ("junk food"). Each receipt and each item carries at MOST ONE category. The scanner's structured-output JSON schema embeds the current taxonomy as an enum, so the LLM assigns categories from MY list at scan time; review corrects. New category = edit the list; next scans pick it up automatically. Categories are live-joined: renaming keeps the stable id (propagates to all history), re-categorizing an old row is reflected on the next fetch of any historical analysis, and deleting a category is ON DELETE SET NULL (referencing rows become uncategorized, live) — not blocked, not snapshotted.
- Tags (decided): a SECOND user-editable taxonomy, ITEM-LEVEL ONLY, MANY-per-item (0..N) — cross-cutting labels orthogonal to the single category (an item categorized "groceries" can be tagged "japan-trip" + "work-reimbursable"). Like categories, the current tag list is embedded as an enum in the scanner's per-item output schema, so the LLM PROPOSES tags at scan time; tags are best-effort (item-level, no accuracy gate) and reviewer-correctable. New tag = edit the list; next scans pick it up. Deleting a tag removes its associations (ON DELETE CASCADE on the join, live); renaming keeps them (stable id).
- Dedupe (decided flow): dedupe is not a separate flow — it's a match panel inside the new receipt's review screen, the screen every scanned receipt already passes through. ONE tolerance-band matching rule surfaces candidates (decided — an earlier "exact total for has-image, band for imageless" split was rejected on re-review: it created an OCR-variance blind spot where two photos of one receipt whose scans differ by a cent, and whose hashes also differ, would never surface as a possible duplicate). The panel is advisory (nothing auto-executes), so surfacing should ERR TOWARD showing a candidate; the has-image vs imageless FLAVOR governs only the offered ACTION, not the match tightness.
    - Match rule (all candidates): same currency; total within the TOLERANCE BAND abs(total − candidate.total) ≤ max(match_total_abs_cents, round(match_total_pct × candidate.total)) (config defaults match_total_abs_cents=200 / $2.00 and match_total_pct=0.05 / 5%); purchase_date within ± the dedupe day window; fuzzy merchant. Candidates are ALL matches ranked by match strength (monotonic in closeness — an exact-total same-day same-merchant hit ranks top), in two labeled flavors:
    - WITH image = "possible duplicate" (both images side by side; actions: discard this upload / keep both). The band (not exact-only) means an OCR-variance re-photo of one receipt still surfaces; a rare false pair (two similar coffees) just shows an extra suggestion the user dismisses with keep-both — cheap, since nothing auto-executes.
    - imageless (pending/confirmed recurring, reviewed manual entries) = "matching transaction" (actions: merge into it / keep both). The band is what lets a variable utility bill (recurring pending row created at expected_amount_cents, real invoice differs) or an approximately-typed manual entry still match — the whole point of "upload the real invoice for a pending recurring charge, dedupe offers the merge".
    - NOTHING executes without an explicit click — the suggested action is only visually highlighted. Merge uses verify-scan semantics (fill empty fields only, never overwrite; the upload's existing scan is reused, no re-scan), mismatches resolved inline in the panel. Merge mechanics (decided): image_key + image_sha256 + raw_ocr_json move to the target, scanned items copy over only if the target has none, any inbound references to the source (offsets_receipt_id, recurring_id, purchase_receipt_id on other rows) are repointed source→target, then the source row is deleted (the image object is retained — ownership transfers). Discarding a duplicate deletes both the row and its stored image object. The upload page's live status trace shows "⚠ possible match found" when the scan lands, deep-linking to review — near-immediate feedback without blocking upload (field matching physically can't run at upload time; it needs scan output).
- Exact-duplicate upload check (decided): receipts store image_sha256 (of the normalized image; plain index, not unique). POST /receipts computes it after normalization and, on a hash hit, returns a conflict with the existing receipt ("are you sure? [shows existing]") WITHOUT creating a row; client re-submits with force=true to proceed. The only dedupe check that can run at upload time; also saves a wasted scan per double-tap. Cross-libvips-version hash drift just degrades to the field-match backstop.
- I can re-open and edit a reviewed receipt later without it losing reviewed status, and delete any receipt (with confirmation).

## Recurring
- I can create/edit/pause/delete recurring transaction definitions: name, merchant, expected amount + currency, cadence = custom interval (decided: every N days/weeks/months/years, anchored to a start date — covers weekly through annual and odd billing cycles; calendar arithmetic with month-end clamping, see Recurring interval semantics), default payment method/category/tax-deductible.
- Recurring definitions can be finite (decided): optional end_date or occurrences_total (a fixed target count, not a decrementing counter — see schema); auto-deactivates when exhausted. Installment plans (Afterpay/Zip: 4 fortnightly payments) are finite recurrings with occurrences_total=4 and an optional link to the originating purchase receipt.
- Recurring definitions carry a kind [purchase|transfer|income] (decided; income added 2026-07-18): auto-savings and credit-card autopay are recurring transfers; salary/paychecks and monthly bank interest are recurring INCOME (inflow). Generated rows inherit the kind + direction — transfers/income stay out of spend stats; income feeds income stats. The generation engine (idempotent ordinal, catch-up, finite ends) is kind-agnostic, so inflows flow through unchanged: a recurring salary auto-creates a pending income row on payday that I confirm (= review), same as any recurring charge.
- Dedupe matching against a recurring pending row is tolerant on BOTH date (± a few days, billing date drifts) AND total (the tolerance band above — the actual charge differs from the expected amount, esp. variable utilities). This is why imageless candidates never use exact-total matching (see Dedupe).
- On the expected date a pending receipt is auto-created. Confirming = reviewing it (adjusting the actual amount if it differs); if the charge didn't happen I delete the pending receipt; if I cancelled the subscription I pause/delete the definition.
- If I upload the real invoice for a charge that already has a pending recurring receipt, dedupe offers the merge.
- Direct attach with verify-scan (decided): from any imageless receipt (source=manual or recurring, image_key NULL) I can attach an image — POST /receipts/:id/image, same normalize→store pipeline as upload, sets image_key and scan_state=uploaded; the exact-dup hash check runs here too, mirroring upload semantics (decided): on a hash hit the attach is rejected with a conflict {duplicate_of: S} before anything is stored; the client dialog shows S and offers (a) re-submit force=true to attach anyway (stores an independent second copy under its own image_key — the sha index is non-unique precisely to permit this) or (b) merge instead — deep-link to S's review screen with the match panel open and this receipt force-included as a pinned candidate (?suggest= on the matches endpoint; field matching alone might not rank it if the entered fields are off). The merge is the standard merge-from with the imageless receipt as TARGET (its user-entered fields are the trusted ones) and S as SOURCE (donates image/ocr/items, then deleted — also un-double-counts stats if both were reviewed). Direction is forced: the imageless row would donate nothing as a source. The existing worker scans it (kind-aware, using the row's known kind). Worker mode discriminator (decided): mode is chosen by TRUST, not provenance — reviewed_at IS NULL → extraction mode (scan writes structured fields); reviewed_at IS NOT NULL → verify mode (fill-empty-only + compare, below). (`source` no longer gates this: a source=scanned row that scan-failed and was then manually filled has reviewed_at set, so a retry VERIFIES rather than clobbering the typed values — see Invariants, trust-gated scan mode.) Scan is verification, not extraction: it NEVER overwrites user-entered values — it fills only empty fields (items, store_name_raw, raw_ocr_json), then compares scanned vs recorded fields (total exact; date ± dedupe tolerance; merchant fuzzy). Review/stats state is DERIVED from timestamps, not a status column (see Invariants): a completed scan stamps proposed_at, and needs_review / counts_in_stats fall out of proposed_at vs reviewed_at. All match → the scan re-stamps reviewed_at (≥ proposed_at) so the row stays reviewed silently, no queue trip; any mismatch → proposed_at now leads → the row is needs_review with the diff shown at review. A never-reviewed row prefills and becomes needs_review as normal. Terminal scan failure touches NEITHER timestamp: a previously-reviewed row stays reviewed (attach degrades to evidence-only) — a confirmed transaction can never land in scan_failed over an attach glitch. Because proposed_at is stamped only when a scan COMPLETES, a reviewed receipt never leaves stats during an in-flight re-scan — even across a process crash and the full reclaim window (the old "brief stats blip" is designed out). Upload-first dedupe→merge remains available.

## Browse
- "Upload history": all transactions, chronological, filterable by kind (purchase/refund/reimbursement/income/transfer), direction (inflow/outflow), state (scan_state and/or the derived needs_review/reviewed), date range, merchant, receipt/income category, item tag, payment method, tax-deductible, source (scanned/manual/recurring).
- Receipt detail view with zoomable image.

## Insights (all decided; confirmed rows only (is_confirmed) — see Invariants; never summed across currencies). SPEND insights select through v_countable_spend (spend-kind allowlist + no stored-value-paid purchases); INCOME insights through v_income; the RECONCILIATION insights read the base v_receipt_state and slice by `direction` — see the view layering in the schema.
- Monthly spend trend: total reviewed spend per month.
- Category breakdown: spend by category for a selected month/range.
- Filtered spend trend: the monthly trend narrowed by dimension selections — receipt category, merchant, and/or item category (e.g. "junk food over time"). Item-category trends aggregate item amounts, not receipt totals, and are best-effort (auto-labeled).
- Income trend + income-by-category (decided M1): total income per month and a breakdown by income category (salary / interest / dividends / other-income) — via v_income. Separate axis from spend, never netted into it.
- Net cashflow (decided M1, falls out for free): income − spend per month (the two trends overlaid), so I can see whether I'm net saving. Transfers excluded from both sides.
- Tax-deductible running total for the current Australian FY (Jul 1 – Jun 30, decided), with its receipt list. All "this year" filters use AU FY. (Income is recorded but M1 makes no attempt at income-tax/PAYG reckoning — that's payroll accounting, out of scope; the tax view stays the deductible-expense list.)
- Payment-method crosscheck: monthly totals per payment method to reconcile against bank statements — NOW TWO-SIDED (decided M1, income in scope): debits (outflows) AND credits (inflows: income, refunds, reimbursements, inbound transfers), matched by `direction`. The M1 reconciliation goal is a full balance on BOTH sides, not debit-only.
- Cash view: cash withdrawn (transfers) vs cash spent per month — the reconciliation story for statement ATM lines.
- SPEND stats follow the spend rules via v_countable_spend (signed_spend_cents, stored-value exclusion); INCOME stats via v_income; reconciliation slices the base view by direction (that's its job — see view layering).
- FX note: foreign-currency receipts never exactly match the AUD card charge (rate + fee); reconciliation for those is approximate by design.
- CSV export of receipts + items for a date range, for tax time.

# Architecture & Stack (decided)

- Backend: C++ (chosen deliberately as a learning vehicle for the language; not the "natural" fit for a CRUD-heavy app, but acceptable given low scale/low risk)
    - Architecture: single all-async Drogon process, coroutines throughout (decided — maximizes C++ async learning; no separate worker binary)
    - Framework: Drogon (batteries-included: routing, JSON, async/coroutines, ORM-lite) to avoid hand-building HTTP plumbing
    - DB access: Drogon's async DbClient with coroutines (`co_await execSqlCoro`). libpqxx rejected: blocking, would stall the event loop
    - JSON: nlohmann/json
    - Outbound HTTP (to scan provider): Drogon's async HttpClient (`sendRequestCoro`). cpr/Boost.Beast rejected: blocking / too low-level
    - Known tradeoffs to budget for: async orchestration of the scan pipeline (upload -> background job -> status transitions) is more manual than in Python/Node; less JSON/schema-validation/ORM convenience; slower iteration (compile times, verbose error handling); no native LLM/OCR SDKs so provider calls are hand-rolled REST+JSON.
- Money: integer cents end-to-end (decided) — BIGINT in Postgres, int64 in C++, plain integers in JSON. Avoids float drift through nlohmann/json doubles; makes dedupe total-matching exact. Frontend formats for display. Amounts are stored as a NON-NEGATIVE MAGNITUDE (≥ 0); cashflow direction is the separate `direction` field, not a sign on the amount (decided 2026-07-18 — see Transaction kinds). Sub-cent upgrade path (if ever needed for unit prices like fuel $1.859/L or a future finer-grained domain): RESCALE the integer unit (e.g. store micro-dollars, 1e-6, in the same int64) — never switch to floats or NUMERIC. Keeps exact arithmetic + exact-match dedupe. Whole receipt totals/line amounts are always whole cents (that's what you're charged), so this stays a documented escape hatch, not an M1 change.
- Database: Postgres
- Object storage (decided): self-hosted MinIO (S3-compatible), running in Docker Compose on the private network, never publicly exposed
    - Access from C++: hand-rolled SigV4 client over Drogon's async HttpClient (PutObject/GetObject/DeleteObject only; ~150 lines of OpenSSL HMAC). AWS SDK rejected: huge build tax, own thread model, 99% unused. minio-cpp rejected: blocking.
    - Cloud-portability requirement (decided): the S3 client takes endpoint, region, credentials, and addressing style (path-style for MinIO, virtual-hosted for AWS) from config — swapping MinIO for AWS S3/R2/B2 later is a config change plus `rclone sync`/`mc mirror` of the objects. ImageStore interface above it keeps even non-S3 backends possible.
    - Image normalization (decided): server-side canonical version regardless of what the client uploads — decode → apply EXIF orientation → STRIP all EXIF (phone photos embed GPS) → downscale to 2048px long edge → encode WebP q80 (~150-350KB/receipt, ~0.5GB/yr; crisp above Haiku's 1568px vision cap, headroom for future models). Original discarded. Library: libvips.
    - CPU-bound rule: normalization runs on a small worker thread pool, resuming the coroutine after — never on a Drogon IO loop.
    - Serving (decided): backend proxy — GET /receipts/:id/image streams from MinIO through Drogon, guarded by the session cookie. Presigned URLs rejected for now: would force MinIO onto the public internet and re-tie the frontend to whichever storage host is in use, breaking swap-invisibility.
    - Images are immutable after normalization → proxy responds with `Cache-Control: private, max-age=31536000, immutable`; the browser cache is the only cache that matters at one user.
    - CDN note / upgrade path: proxying does forgo CDN caching of a future cloud bucket — irrelevant here (private per-user content, view-once access pattern, audience of one). If that ever changes, the documented upgrade is the same endpoint 302-redirecting to a short-lived presigned URL: auth stays in the backend, frontend unchanged, CDN/edge takes over delivery. ~20-line change inside the same handler.
- Scan pipeline: async, decoupled from upload request
    - Core invariant (decided): a receipt row is created upfront the moment work exists (upload, manual entry, recurring charge date) so no intermediate "waiting" state can lose work. All non-manual receipts require human review before counting toward stats/analytics — stats queries filter on the derived counts_in_stats predicate (reviewed_at present and at least as recent as any machine proposal; see Invariants), never on a raw status column.
    - POST /receipts saves image + creates receipt row (scan_state=uploaded), returns immediately
    - Background worker: in-process event-loop timer (`getLoop()->runEvery`) polling the scan_state column (no Redis/BullMQ at personal scale). Sets scan_state=scanning, calls scan service. Cap concurrent scans (~2-3).
    - Stuck-scan recovery (required, since in-flight scans die on process restart): receipts get scan_attempts + reclaim_count + scan_started_at; worker reclaims rows stuck in scan_state=scanning past a timeout. TWO separate budgets so deploy churn can't fail a good receipt (see Invariants): scan_attempts counts only real provider/parse errors (cap = 3 → scan_failed); reclaim_count counts crash/timeout reclaims (high backstop cap for genuine runaway hangs). A reclaim re-queues (scan_state=uploaded) WITHOUT spending a scan_attempt — "we never got an answer" is not evidence the scan is bad, and it must not pollute the <5% hard-failure quality metric.
    - Atomic job claim: worker claims jobs via `UPDATE ... WHERE id IN (SELECT ... WHERE scan_state='uploaded' FOR UPDATE SKIP LOCKED) RETURNING`, so the process stays stateless (DB is the queue's source of truth) and a second instance could never double-claim. Statelessness invariant: all in-process state (in-flight coroutines, timers, counters) is disposable and reconstructible from Postgres + MinIO.
    - Scan service (decided): multimodal LLM extraction, launch provider = Claude Haiku 4.5 (`claude-haiku-4-5`) via the Claude API with structured outputs (JSON-schema-constrained response) — ~0.5c/scan, guaranteed-parseable JSON.
    - Document-type classification (decided, M1 income): the scan is kind-aware — it first classifies the document (purchase receipt | return docket → refund | payslip/paycheck → income) and sets kind + direction accordingly, then extracts the fields appropriate to that type. Purchase/refund → merchant, date, total, items, categories/tags. Payslip → employer (store_name_raw/merchant), pay date, NET pay as the income `total_cents` (magnitude, direction=inflow), income category (e.g. "salary"); gross/tax/super lines may be captured as best-effort informational items but NEVER feed stats/reconciliation (super boundary). The output JSON schema carries a `doc_type`/`kind` discriminator; category enums are context-appropriate (expense categories for receipts, income categories for payslips — both drawn from the user's own taxonomy via the is_income flag). One adapter, one schema with a discriminated shape — keeps the ScanProvider boundary unchanged.
    - Payslip input note: paychecks are often digital PDFs; the M1 image pipeline (JPEG/PNG/WebP, no PDF rasterizer) expects an IMAGE, so a digital payslip is captured by SCREENSHOT (works today). Native PDF ingestion is out of M1 scope (would add a rasterizer dep — see Open Eng Decisions).
    - HARD REQUIREMENT — provider swappability: the scanner controller must make it trivially easy to swap any model from any LLM provider. Design: abstract `ScanProvider` interface (`scanReceipt(imageBytes/imageKey) -> ScanResult`), one adapter class per provider (Anthropic first; OpenAI/local/etc. later). Each adapter owns its own request shape, prompt, schema enforcement, and response parsing; the rest of the app only ever sees the normalized ScanResult struct. Provider + model + endpoint + API key come from config, not code, so a model swap within a provider is a config change and a new provider is one new adapter class.
    - On success: scan_state=idle + proposed_at stamped, structured fields + items written → the row derives as needs_review
    - On failure: scan_state=scan_failed, user can retry or enter manually
    - Frontend polls for status updates (simplest option at this scale; SSE/websocket possible later)
- Recurring transactions (decided): when a recurring charge's expected date arrives, the worker immediately creates a receipt row (scan_state=idle, source=recurring, recurring_id FK, inheriting the definition's kind + direction, proposed_at stamped + reviewed_at NULL so it derives as needs_review). Works identically for outflows (charges, transfers) and inflows (recurring income like salary/interest). The row itself is the confirmation-queue entry: confirm = mark reviewed, cancelled/paused = delete. No separate recurring_charges table. Dedupe should offer to merge an uploaded receipt image into a matching pending recurring receipt rather than flag a duplicate.
- Recurring interval semantics (decided): month/year units use calendar arithmetic, not fixed day counts — monthly = (1, month), quarterly = (3, month), yearly = (1, year). Occurrence k is always computed from the anchor (anchor + k×interval), never from the previous occurrence, so clamping can't drift. Month-end clamping: if the anchor's day-of-month doesn't exist in a target month, clamp to that month's last day (Jan 31 → Feb 28/29 → Mar 31; Feb 29 anchor → Feb 28 in non-leap years) — matches Stripe/RFC 5545 behavior. All dates in Australia/Sydney. Real-world billing wobble (±days) is handled at review/dedupe-merge, not by the scheduler. Occurrence bookkeeping (decided): the definition stores occurrences_generated; occurrence k = that counter, so next_expected_date = anchor + (occurrences_generated+1)×interval (clamped) — the ordinal is tracked EXPLICITLY, not re-derived from the clamped date (which isn't cleanly invertible once month-end clamping applies). Generation is idempotent via a unique (recurring_id, occurrence_ordinal) on the generated receipts, so a double timer tick or a crash between insert and counter-advance can't double-create. Catch-up after downtime (decided): if several occurrences elapsed while the process was down, generate ONLY the latest missed occurrence (one receipt row) but ADVANCE occurrences_generated by the full elapsed delta to the latest elapsed ordinal — bills don't want N identical stale pending rows, yet the ordinal must still reflect reality so finite plans deactivate correctly. Skipped ordinals simply never get a row (the unique (recurring_id, occurrence_ordinal) permits gaps); a finite plan that slept through installments 2-3 resumes at 4 with occurrences_generated=4, so it still ends after the right number of cycles rather than running 2 cycles long. Single source of truth for "how far along" is occurrences_generated — see the finite-condition note in the schema (occurrences_total, not a separately-decrementing counter, to avoid two counters drifting).

First-party cookies are not affected by browser third-party-cookie blocking. Genuine multi-tenant (users table; ALL per-user rows scoped by user_id; ownership check on every route) — every signup is an equal peer, no owner/admin role, no first-user special case. Only-user-today is incidental.

Intended (matches the live-join philosophy used for categories); flagged here so it isn't a surprise. If per-receipt exclusion ever needs to be frozen at record time, snapshot the exclusion bit onto the receipt (also noted in Invariants).

# Invariants (load-bearing — each gets a test)

Cross-cutting rules multiple features depend on. They're easy to violate in code, so they're stated once here and each is covered by a test.

- **Zero-loss:** a receipt row exists the instant work exists (upload, manual entry, recurring due-date). No in-memory-only state can lose work. All in-process state (coroutines, timers, counters) is disposable and reconstructible from Postgres + MinIO.
- **Two-axis state (there is NO `status` column):** a receipt's lifecycle is two orthogonal axes.
    - scan_state[idle|uploaded|scanning|scan_failed] — the scan-pipeline lane ONLY. idle = at rest (manual, recurring, or scan-complete); uploaded = queued for the worker; scanning = in flight; scan_failed = terminal-but-retryable.
    - review/stats state — DERIVED from two timestamps, never stored:
        - proposed_at = last time a machine proposed field values (scan completion OR recurring auto-creation).
        - reviewed_at = last time a human confirmed the row.
        - needs_review ⇔ proposed_at IS NOT NULL AND (reviewed_at IS NULL OR proposed_at > reviewed_at)
        - counts_in_stats ⇔ reviewed_at IS NOT NULL AND (proposed_at IS NULL OR reviewed_at ≥ proposed_at), further filtered by the stats rules (kind ≠ transfer; not a stored-value-paid purchase).
    - Single-source predicate (layered): the timestamp membership (needs_review, is_confirmed) is defined ONCE in the SQL view `v_receipt_state` (migration 001), never re-inlined. Spend membership (`counts_in_stats` = is_confirmed AND kind IN (purchase,refund,reimbursement) AND not a stored-value-paid purchase — an explicit spend-kind allowlist, NOT "≠transfer", so income is not swept in) is `v_countable_spend` (which also carries the signed_spend_cents column); income membership is `v_income` (is_confirmed AND kind=income); reconciliation stats reuse the base v_receipt_state because they NEED the transfer / income / stored-value rows and slice by `direction`. The Invariants tests target the views; every stats/queue endpoint selects through the appropriate layer (see schema).
    - Designed-in consequence (this REPLACES the old "stats blip"): proposed_at is stamped only when a scan COMPLETES, so a reviewed receipt being re-scanned (attach/merge) stays in stats through the whole in-flight window and even across a crash + reclaim; it leaves stats only at the instant a completed scan finds a mismatch.
- **Trust-gated scan mode:** reviewed_at IS NULL → extraction (scan writes fields); reviewed_at IS NOT NULL → verify (fill-empty-only + compare, never overwrite). Provenance (`source`) does NOT gate this — a manually-filled scan-failed row is verify-mode on retry.
- **Scan-attempt vs reclaim budgets:** scan_attempts counts only real provider/parse errors (cap 3 → scan_failed); reclaim_count counts timeout/crash reclaims (high backstop cap). A reclaim re-queues WITHOUT spending a scan_attempt, so deploy churn can't fail a good receipt or pollute the <5% hard-failure metric.
- **Recurring generation is idempotent:** unique (recurring_id, occurrence_ordinal); ordinal tracked via occurrences_generated (the single mutable progress counter); catch-up generates only the latest missed occurrence but advances occurrences_generated by the full elapsed delta, so a finite plan (occurrences_total target, not a decrementing counter) still ends after exactly N cycles even after downtime.
- **Equal-tenant registration:** registration is open self-service; every account is an equal isolated peer (no owner/admin role, no first-user privilege). Creating a user seeds THAT user's default categories + "cash" payment method in the same transaction. Email is unique per user (case-insensitive); no global state distinguishes one account from another.
- **Referential cleanup:** inbound receipt FKs (offsets_receipt_id, recurring_id, purchase_receipt_id) and reference-data FKs (category_id, payment_method_id) are ON DELETE SET NULL; item_tags is ON DELETE CASCADE; merge repoints inbound receipt FKs source→target before deleting the source. Accepted consequence: deleting a stored-value payment method un-excludes its past purchases from spend (they re-enter stats under "no payment method") — payment methods aren't meant to be deleted; if this ever bites, snapshot just the exclusion bit onto the receipt.
- **Tenant isolation (load-bearing, now that multi-user is real not hypothetical):** every resource query carries the user_id predicate; non-owned/missing id → 404 (no existence leak). No endpoint can read, join, or mutate across tenants. The image_sha256 dup check is scoped (user_id, image_sha256); the scanner's category/tag enums are built from the requesting user's own taxonomy. A cross-tenant read is a security bug, not a nuisance — each gets a test.
- **Money:** integer cents end-to-end, stored as a NON-NEGATIVE magnitude (direction is the separate `direction` field, never the amount's sign); per-kind stat signs live in the views (signed_spend_cents), not in storage; dedupe/match compares magnitude within a single currency; stats group by currency, never sum across. Spend membership is a spend-kind ALLOWLIST (purchase/refund/reimbursement), not "≠transfer" — income must never leak into spend.

# Success Criteria

Priority (decided): **daily-use tool first**. Success = still using it in 6 months. When instructive-but-slow conflicts with shipping, ship — C++ learning is pursued opportunistically within that constraint, never as a reason to stall a feature.

## Product — the app is genuinely used
- Capture bar (decided): EVERY transaction — receipts, cash, cards, subscriptions, AND income (paychecks, interest, distributions). The app is the complete money record (both sides), not just spending; bank reconciliation should fully balance on debits AND credits, not approximately.
- Habit: after M1 ships, transactions are captured within 3 days of purchase, sustained for 3 consecutive months without reverting to spreadsheets/nothing.
- Friction ceilings (these make the every-transaction bar survivable): recording a cash transaction <15s (implies a quick-add form with smart defaults, not a full page of fields); typical scanned receipt reviewed in <30s; pending-review queue hits zero at least weekly.
- Zero-loss invariant holds in practice: no receipt ever disappears into a waiting state; every upload is accounted for as reviewed, pending, or failed-with-image.
- Trust: monthly payment-method totals reconcile against bank statements fully on BOTH sides — debits (spend/transfers out) AND credits (income, refunds, reimbursements, transfers in), matched by direction (income is in M1 scope now); discrepancies explainable in minutes, not hours. (Off-bank flows — employer super SG, salary-sacrifice, gross/PAYG — are intentionally excluded: only bank-touching amounts are tracked.)
- Tax time test: at FY end, producing the deductible receipt list + CSV export takes <5 minutes.

## Scanner quality (measured against the golden set)
- Golden set: 20-30 real receipt photos (varied: faded thermal, crumpled, long dockets, glare) with hand-verified expected values, kept as fixtures, PLUS a handful of payslip/paycheck images (income scanning is in M1) — different layout, so a few fixtures guard the doc-type classifier + net-pay extraction. Re-run on every prompt/model/provider change — the regression harness that makes provider swapping safe, not just easy.
- Field accuracy targets (proposed): receipts — total ≥98%, purchase date ≥95%, merchant ≥90%, receipt category ≥80%; payslips — doc-type classified correctly ≥98%, NET pay amount ≥98%, pay date ≥95%, employer ≥90%. Items / gross-tax-super breakdown best-effort, no gate.
- Hard scan failures (scan_failed after retries) <5% of uploads.

## Engineering & ops
- Restart-safe: killing the process at any moment loses no data; stuck scans self-recover via the reclaim rule (verified by test).
- Deploy is one command (docker compose up on the VPS); a fresh-machine rebuild from repo works, proving the vcpkg/CMake setup is reproducible.
- Backups: Postgres + images backed up automatically; a restore drill has actually been performed once (an untested backup doesn't count).
- Security floor on the public VPS: TLS via Caddy, rate-limited login AND registration, argon2 hashing, open signup guarded by email verification (spam + LLM-cost-abuse gate), strict per-tenant isolation (cross-tenant read = failing test), no secrets in repo.
- Running cost: (proposed) <$15/month all-in (VPS + LLM; LLM share expected <$1).

## C++ learning outcomes (decided: all four, subordinate to shipping)
- Async & coroutines: can add a new endpoint end-to-end (route → coroutine handler → async SQL → JSON response) without copying from references; can explain why blocking the event loop is wrong and demonstrate the failure mode.
- Ownership & RAII: no raw owning pointers in the codebase; can justify each lifetime/ownership decision (value vs unique_ptr vs shared_ptr) rather than defaulting to shared_ptr everywhere.
- Build & tooling: owns the CMake + vcpkg setup rather than cargo-culting it; dev builds run with warnings-as-errors and ASan/UBSan; fresh-machine rebuild from repo works.
- Testing discipline: unit tests (Catch2 or GoogleTest) + integration tests against Postgres are a habit, not an afterthought; FakeScanProvider exercises the review pipeline without LLM calls; tests run before every deploy.
